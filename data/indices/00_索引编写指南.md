## 三、简述编写标准

### 3.1 核心要求：两必填 + 一可选

一条合格的简述**必须**覆盖前两个要素，第三个**有则写、无则空**：

| 要素 | 必填 | 定义 | 检查方法 |
|------|:----:|------|---------|
| ① **是什么** | ✅ | 物种/形态/身份/功能类型 | 读完知道这是个什么东西 |
| ② **核心能力** | ✅ | 1-2个关键能力词 | 读完知道它能干什么 |
| ③ **叙事钩子** | — | 为什么有趣/值得用的一个亮点 | 读完想说"这个有意思"；**并非所有妖怪都有钩子，没有就不写** |

> **判定规则**：缺①或②→不合格；仅缺③→合格（③是加分项，不是扣分项）

### 3.2 判据示例

| 评级 | 简述 | 分析 |
|:----:|------|------|
| ❌ 不合格 | "千年白猿" | 只有①，缺②③ |
| ❌ 不合格 | "水族精怪的经典类型" | 太模糊，①②③全缺——**评价代替描述** |
| ❌ 不合格 | "唐传奇中最经典的器物成精形象之一" | **评价代替描述**：没说它是什么、能做什么 |
| ❌ 不合格 | "乌江一带的蛇域统领" | **空壳简述**：只有定位短语，无能力无叙事 |
| ❌ 不合格 | "地狱关中的刑吏" | **极简残缺**：5个字的身份，无任何细节 |
| ❌ 不合格 | "非羊也，雨工也" | **残缺引文**：只有一句引文，读完不知道它是什么能做什么 |
| ✅ 合格 | "蚯蚓王，白颈红斑，率领数百小蚓" | 有①②，缺③但③非必填——合格 |
| ⚠️ 勉强 | "龙宫家族，龙母+至少两子一女的完整家庭结构" | **组织描述型**：有①但缺②，没说这个家族能干什么 |
| ✅ 合格 | "千岁白猿，人形时为美髯白衣丈夫，原形为白毛大猿。力能杀人、浑身如铁、半日往返数千里、通晓符篆、双剑环身如月" | ①物种形态 ②多维能力 ③视觉冲击力 |
| ✅ 合格 | "盘踞大冢的巨型猪妖，在当地建祠假冒'乌将军'，每年以人祭少女为食——真身为大猪（缺左前蹄）" | ①身份 ②假冒地方神 ③"每年吃少女"的叙事钩子 |

### 3.3 常见简述病症对照表（2026-04-16 唐传奇审查沉淀）

> 以下六种病症来自对90条唐传奇妖怪简述的全量审查，34条不合格均可归入这六类。

| 病症 | 检测方法 | 修复方法 | 典型反例 |
|------|---------|---------|---------|
| **空壳简述** | 删去原文引号后实质内容<20字 | 回全文层补能力+叙事钩子 | "乌江一带的蛇域统领" |
| **评价代替描述** | 含"经典""典型""代表性""适合"等评价词 | 删评价词，换原文能力描写 | "唐传奇中最经典的器物成精形象之一" |
| **纯景观/纯罗列** | 描述对象不能独立行动 | 补独立行为，或标"场景元素" | "千龙万蛇缠绕成桥柱" |
| **组织描述型** | 简述主语是"族群/家族/体系" | 补1个代表性能力或核心冲突 | "龙母+至少两子一女的完整家庭结构" |
| **残缺引文型** | 简述仅为1句定位+引文 | 保留引文但追加实质描述 | "非羊也，雨工也" |
| **缺①或②** | 有①但缺②（最常见），或有②但缺① | 针对性补充缺失的必填要素 | "龙宫家族，龙母+至少两子一女"（缺②核心能力） |

### 3.4 条目性质预判（审查前必做）

在审查简述之前，先判断条目属于以下哪种，**不同性质的条目简述标准不同**：

| 性质 | 判据 | 简述标准调整 |
|------|------|------------|
| **A 独立妖怪** | 有独立形态+独立行为 | 正常三要素审查 |
| **B 从属/召唤物** | 依附于主体存在（如白猪幼崽群→昭潭三白猪妖） | 降低③叙事钩子要求，重点写"与主体的关系+自身特征" |
| **C 场景元素** | 不能独立行动（如千龙万蛇桥柱） | 不强制要求②核心能力，改为"场景效果描述" |
| **D 族群/组织** | 描述群体而非个体（如南海龙族） | 要求写"群体特征+代表性冲突"而非个体能力 |

> **应用规则**：性质C和D的条目不按标准三要素打❌，但必须在简述中**标注性质标签**——如 `[场景]`、`[组织]`，以便策划理解条目的使用方式。

### 3.5 能力词必填规则

简述中**必须**出现至少1个**动词性能力描述**：
- ✅ "能XX""会XX""擅长XX""可XX"
- ✅ "喷射毒液""化为人形""控制池水""吞食象群"
- ❌ "巨型""千年""经典""十丈"——**纯名词/形容词堆叠不算能力描述**

**例外**：性质C（场景元素）不要求能力词，但需有场景效果动词（"缠绕""支撑""照耀"）。

### 3.6 简述的数据来源

简述**必须**来自全文层 🎮游戏可用元素对应子类的**前半段**（素材描述部分），可以缩写但不可编造。

**不可混入**的内容：
- 🚫 后半段的游戏设计想法（"可设计为…"）→ 那是「游戏设计建议」列的内容
- 🚫 评价性语句（"这是一个经典的…"、"非常适合…"）
- 🚫 跨条目的比较（"类似于MO-010…"）

### 3.7 简述的长度建议

- **最短**：20字（覆盖三要素的极限压缩）
- **推荐**：40-100字
- **最长**：150字（超过则应精简）

## 四、游戏设计建议编写标准

### 4.1 定位

「游戏设计建议」列是**索引层新增字段**（v2.0→v2.1），目标是让L1概览卡完全自包含——策划看索引就能判断"这个东西在游戏里能怎么用"，不需要再回全文层取数据。

### 4.2 数据来源

直接复用全文层 🎮游戏可用元素对应子类的**后半段**（"可设计为…"部分），**原样提取，不重新生成**。

原文示例：
> **千岁白猿**——千年灵猿，人形时为美髯白衣丈夫…（前半段=简述）→ **可设计为一个高等级BOSS——南山白猿王，藏身洞府，掳走女性NPC…**（后半段=游戏设计建议）

### 4.3 编写规则

| 规则 | 说明 |
|------|------|
| **原文优先** | 如果全文层已有"可设计为…"的文字，直接提取 |
| **不可编造** | 不要让AI重新生成设计建议，只做原文的缩写/提炼 |
| **关键词密度** | 每条≥2个具体游戏机制关键词（BOSS/NPC/关卡/任务类型/选择分支/系统设计等） |
| **长度控制** | 20-80字为宜，压缩原文但保留核心设计方向 |
| **无原文时标记** | 如果全文层🎮对应子类没有后半段设计想法，填 `—`（空值），不编造 |

### 4.4 按出处的填写规则

| 出处 | 是否必填 | 说明 |
|------|:-------:|------|
| **唐传奇** | ✅ 必填 | 全文层已有"可设计为…"文字，直接提取填入 |
| **酉阳杂俎** | — 留空 | 当前全文层尚未生成游戏设计建议，统一填 `—`；待后续补充时再更新 |
| **跨文献条目**（出处标`唐传奇+酉阳杂俎`） | ✅ 必填 | 以唐传奇全文层的"可设计为…"为准 |

> **后续规则**：当酉阳杂俎完成深度拆解并生成🎮游戏可用元素后，相关条目的「游戏设计建议」应同步补充。

### 4.5 判据示例

| 评级 | 游戏设计建议 | 分析 |
|:----:|------------|------|
| ❌ 不合格 | "适合做BOSS" | 太笼统，没有具体机制 |
| ❌ 不合格 | "这个精怪非常有趣，建议重点开发" | 评价性语句，不是设计建议 |
| ✅ 合格 | "高等级BOSS·南山白猿王，掳走女性NPC触发救援任务，弱点：铁器+伏击" | 定位+任务类型+弱点=完整设计线索 |
| ✅ 合格 | "隐藏NPC群落：山野中看似普通的一家人，实为虎精伪装；触发条件：夜间探访" | 功能定位+场景+触发机制 |

## 五、精彩度评级标准

### 5.1 三级制

| 等级 | 标准 | 判据 |
|:----:|------|------|
| ⭐⭐⭐ **直接可用** | 外形清晰 + 能力独特 + 有弱点/破解 + 有场景 + 有叙事钩子 | 5项中满足4-5项 |
| ⭐⭐ **需要补充** | 有基本信息但缺1-2个维度 | 满足2-3项 |
| ⭐ **仅供参考** | 太模糊/纯博物条目/缺乏游戏化抓手 | 满足0-1项 |

### 5.2 五项判据详解

| 判据 | 合格标准 | 不合格示例 |
|------|---------|----------|
| **外形清晰** | 读完能在脑中形成画面 | "水族精怪的经典类型" |
| **能力独特** | 有≥1个可转化为游戏机制的能力 | "盘踞水底洞穴"（没说能做什么） |
| **有弱点/破解** | 记录了弱点或克制方法（如有） | — |
| **有场景** | 简述中暗示了适用场景（地点/时间/情境） | — |
| **有叙事钩子** | 能引出冲突/悬念/反转/情感 | "普通的蛇精" |

### 6.1 标点统一

| 场景 | 正确 | 错误 |
|------|------|------|
| 母题类型分隔 | `A(复仇/报应/正义)  ·  C(变形/变身/精怪)` | `A(复仇/报应/正义), C(变形)` |
| 原文引用 | `"如股，长丈余"` | `「如股，长丈余」` |
| 破折号说明 | `——` （全角双破折号） | `--` 或 `—` |
| 空缺字段 | `—` | 空白 / `无` / `N/A` |

### 6.2 出处索引表格式

每个索引文件末尾应附**按出处查**的第二索引表：

```
| 故事ID | 篇名 | XX数 | XX名称 |
|:------:|------|:------:|--------|
```

## 七、数据提取规则

### 7.1 提取源

| 字段 | 提取自 | 定位方法 |
|------|-------|---------|
| 名称 | 全文层 🎮游戏可用元素 · 对应子类 | 子类标题下加粗文本（如 `**板桥三娘子**`） |
| 简述 | 同上 · 前半段 | 加粗名称后的描述文字，到"可设计为"之前截止 |
| 游戏设计建议 | 同上 · 后半段 | "可设计为…"开始的文字 |
| 母题类型 | 全文层 🧬叙事母题 + 母题分类总表 v3 | 按 v3 体系标注 |
| 精彩度 | 人工标注 | 按§五的五项判据评级 |

### 7.2 脚本提取防错清单

> 以下规则来自实际踩坑经验，**每次生成索引脚本都必须逐项检查**。

| # | 检查项 | 说明 |
|---|--------|------|
| 🔴1 | **字段标题排除** | 提取结果中不能包含 `妖怪/精怪`、`法术/能力`、`NPC/神灵` 等字段标题本身 |
| 🔴2 | **Section边界** | 提取范围严格限于 `**XX：**` 和下一个 `**YY：**` 之间，遇到 `###`/`---`/`🎭`/`📚` 立即停止 |
| 🔴3 | **黑名单过滤** | 关键词黑名单：`总体评价`、`后世传播`、`后世影响`、`精彩叙事桥段`、`NPC/神灵` |
| 🟡4 | **角色表辅助扫描** | 除🎮字段外，同时扫描 `👤角色档案` 中身份含 `龙/狐/精/妖/鬼/仙/神/灵` 的行 |
| 🟡5 | **跨文件去重** | 已知同源故事需合并（见§七.3），用名称模糊匹配找疑似重复 |
| 🟢6 | **分类审查** | 分类依据是"主体是什么物种/存在形态"，不是"出现在什么场景中" |
| 🟢7 | **名称清洗** | 检查所有名称列，确认无字段标题/评价段落混入 |
| 🟢8 | **简述与游戏设计建议分离** | 简述不混入"可设计为…"，游戏设计建议不混入素材描述 |

### 7.3 脚本合并/重建防错清单（2026-04-16 妖怪索引v2.0合并沉淀）

> 以下规则来自妖怪精怪索引v1.3→v2.0合并时的实际事故，**每次用脚本做索引合并或重建时必须逐项检查**。

#### 7.3.1 文件读写安全

| # | 检查项 | 说明 | 事故来源 |
|---|--------|------|---------|
| 🔴1 | **禁止INPUT=OUTPUT同一文件** | 脚本的输入文件和输出文件**必须不同**。即使脚本在内存中处理完毕，若写入前（如print阶段）崩溃，文件已被截断为空——数据不可恢复 | v2.0合并：脚本先open(w)清空了文件，后在print时GBK编码崩溃，文件变空 |
| 🔴2 | **先写临时文件再替换** | 写入 `{filename}.tmp` → 验证行数/条目数 → `os.replace()` 覆盖原文件。任何异常都不会破坏原文件 | 同上 |
| 🔴3 | **执行前备份** | 脚本开头做一次 `shutil.copy2(input, input + '.bak')`，确保有回退路径 | v2.0合并：因无git也无备份，数据丢失后只能从旧版v1.2重建 |
| 🟡4 | **PowerShell编码陷阱** | Python脚本在Windows PowerShell下运行时，`print()` 默认走GBK编码。如果输出含emoji（⭐/🎮等），必须在print之前加 `sys.stdout.reconfigure(encoding='utf-8')` | v2.0合并：print星星emoji导致UnicodeEncodeError，脚本崩溃 |

#### 7.3.2 分类插入逻辑

| # | 检查项 | 说明 | 事故来源 |
|---|--------|------|---------|
| 🔴5 | **新条目必须插入到表格数据行末尾** | 当检测到下一个分类标题（`### XX类`）时，新条目应追加到**上一个分类的最后一行数据之后**、**下一个分类标题之前的空行之前**——而不是直接塞在标题行之前 | v2.0合并：flush时机错误，新条目被插在`### 新分类（N条）`标题的正上方，导致全部偏移一个分类 |
| 🔴6 | **分类section结构必须完整** | 每个分类section的结构是：`### 分类名（N条）` → 空行 → 表头行 → 分隔行 → 数据行若干 → 空行。脚本插入时不能破坏这个结构 | v2.0合并：插入位置在表头和分隔行之间，导致表格渲染失败 |
| 🟡7 | **分类标题中的条目数必须与实际一致** | 脚本最后必须重新统计每个分类section内的实际数据行数，反写到标题的`（N条）`中——不能用预计算的数字 | v2.0合并：标题数字与实际不匹配（花木植物精标题2条但实际应6条） |

#### 7.3.3 数据完整性验证

| # | 检查项 | 说明 | 事故来源 |
|---|--------|------|---------|
| 🔴8 | **脚本结束前必须验证总数** | `assert 实际MO-行数 == 预期总数`，不等则abort不写入 | v2.0：第二次运行因读到空文件，直接写出了0条目的空壳——如果有assert就会拦住 |
| 🟡9 | **ID连续性检查** | 验证MO-001到MO-最大值是否连续无跳号 | — |
| 🟡10 | **旧前缀残留扫描** | 合并后全文扫描旧前缀（如`YY-`），确认0个残留 | v2.0合并：需确认旧`YY-`条目已全部替换为`YYCZ-` |
| 🟡11 | **换行符完整性** | 验证写入后的文件行数 ≥ 条目数×1.5（每条至少占1行+表头/标题/空行），如果远低于预期则说明换行符被吞 | v2.0合并：第一次合并后1086行内容被压缩到约150行 |

## 八、质量审查流程

### 8.1 硬性自检（20项，必过，不过不交付）

| 类别 | 检查项 |
|------|--------|
| **数据完整度** | ✅1 总数核验 · ✅2 遗漏排查 · ✅3 副本同步 · ✅4 跨文献同源标注 |
| **命名准确性** | ✅5 名称来自原文 · ✅6 无标题污染 · ✅7 无评价段混入 |
| **分类合理性** | ✅8 分类边界清晰 · ✅9 非本类排除 · ✅10 异质条目标记 |
| **格式一致性** | ✅11 表头统一 · ✅12 字段齐全 · ✅13 标点一致 · ✅14 无截断 |
| **交叉引用** | ✅15 ID可反查 · ✅16 源文档有对应 · ✅17 出处索引表与主表一致 |
| **版本管理** | ✅18 版本号 · ✅19 日期 · ✅20 变更记录 |

### 8.2 内容质量抽查（至少30%条目）

| 类别 | 检查项 |
|------|--------|
| **策划可用性** | 📋1 外形可视化 · 📋2 能力明确 · 📋3 弱点/破解（如有） · 📋4 场景暗示 |
| **差异化** | 📋5 同类内不重复 · 📋6 独特卖点 · 📋7 等级层次 |
| **叙事价值** | 📋8 有冲突 · 📋9 有悬念 · 📋10 有反转 · 📋11 有情感 |

### 8.3 简述专项审查

对每条简述逐一检查三要素覆盖：

**第一步：条目性质预判**（见§3.4）
```
A=独立妖怪 → 正常三要素审查
B=从属/召唤物 → 降低③要求
C=场景元素 → 不强制②
D=族群/组织 → 要求群体特征+冲突
```

**第二步：必填项打分**（③叙事钩子为可选，有则写无则空）
```
✅ = ①②齐全（③有无均可）
❌ = 缺①或② → 立即修改
```

**第三步：病症分类**（见§3.3）
```
对⚠️和❌条目标记属于哪种病症，以便批量修复
```

审查输出格式：
```
| ID | 名称 | 性质 | ①是什么 | ②核心能力 | ③叙事钩子(可选) | 判定 | 病症类型 | 修改建议 |
```

## 十、与SKILL.md的关系

本指南是索引**编写**层面的规范。SKILL.md中的输出分级协议（L1/L2/L3）是索引**使用**层面的规范。两者的对应关系：

| SKILL.md输出协议 | 本指南对应章节 |
|-----------------|--------------|
| L1概览卡·简述字段 | §三 简述编写标准 |
| L1概览卡·游戏设计建议字段 | §四 游戏设计建议编写标准 |
| L1概览卡·精彩度字段 | §五 精彩度评级标准 |
| L1概览卡·核心能力标签 | 从简述提炼，不在索引层存储 |
| L1概览卡·配套法术 | 跨索引关联（通过故事ID），不在索引层存储 |

**关键原则**：索引层的字段越完整，L1概览卡就越自包含，策划体验越好。v2.1的核心目标就是通过新增「游戏设计建议」列，让L1不再需要回全文层取数据。
